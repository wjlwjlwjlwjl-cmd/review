# ELF文件
## ELF文件的组成部分：
1. ELF Header ELF头
记录着关于ELF文件的各种信息，包括各部分的开始结束位置
2. Program Header Table 程序头表
记录着segment的信息，包括开始位置、偏移量、长度等等
3. section 节
会有多个，记录着同种信息，是ELF文件的基本组成单位，包含了特定类型的数据 
，如数据节存放了全局变量和静态数据，代码节存放了可执行代码
4. section header table，包含对节的描述
## 常见的ELF文件
1. 可执行文件
2. 可重定位文件：linux下xxx.o文件，windows下xxx.obj文件，可以与其它
文件链接生成可执行文件
3. 共享目标文件，常说的动态库，xxx.so文件
4. core.dump文件存放当前进程的上下文，段错误时如果系统允许的话就会产生
## ELF可执行文件的加载
1. 合并section为segment<br>
具有相同类型的节在加载进内存时会被合并在一起，比如只读、只写、可执行，
具体的分类规则被记录在program header table当中
2. ELF文件的链接视图与执行视图<br>
链接时，文件结构的粒度更细，将文件按功能模块的差异进行划分；执行试图
告诉操作系统如何加载可执行文件，如何完成进程内存的初始化.<br>
虽然program header table是一个ELF文件的可选部分，但是对于可执行文件，
一定有program header table，因为这时对操作系统如何执行可执行文件的指示<br>
## 合并section为segment的原因
内存中处理的基本单位是4kb，所以假设现在一个section的大小是4097字节，两一个section的大小是512字节，那么加载进内存是，就需要用三个页来存放这两个section，
但是如果对section进行合并的话，连接器就可以将小的section和大的section进行合并，这样就可以占用两个页，来节省空间；同时具有相同属性的section被集中
存放在一起，比如只读、可读可写、可执行、需要申请内存，这样也方便操作系统在内存中进行权限管理
## 常见的节
1. .text节，存放程序代码指令
2. .data节，存放已初始化的全局变量和局部静态变量
3. .roread节，存放只读常量，例如字符串，在可执行文件中，存放于只读区，
存放于.text段
4. .bss节，为未初始化的全局变量和局部静态变量预留的位置
## 静态链接
在使用反汇编时，对于一些函数，还没有链接静态库时，使用objdump -d xxx.o
查看汇编代码时可以发现，这些call的地址都是0，这是因为没有进行连接，不知道
这些函数应该到什么地方去调用，在链接时，根据.o文件和静态库的从定向表，就可以
填补上这些未知调用地址的全局函数变量，如果依然有没有填上的地址，就会连接报错
## elf文件的统一编址
在elf文件中，对代码和数据都进行了统一编址。虚拟地址在还没有加载到内存的时候
就已经把可执行程序进行统一编址了。在创建内核数据结构的mm_struct时，反而是通过
可执行文件segment中的信息来进行初始化[start, end]等数据范围<br>
所以虚拟地址，不止需要操作系统支持，也需要编译器支持<br>
elf可执行文件中，也记录了程序的入口，使用readelf -d ...查看
## 动态链接
静态链接会在链接时确定函数变量的调用地址，而动态链接在程序加载到内存时
才进行这一过程<br>
首先，main函数其实并不是c/c++程序的入口，而是_start。调用_start时，
会初始化程序的堆栈结构，随后从程序的数据段拷贝已初始化的全局变量
和局部静态变量到相应内存位置，并清零未初始化数据<br>
动态链接器会自动检测需要动态链接的库，确定其被拷贝到内存当中，这样一旦
动态库被加载到内存当中，就以根据其在内存中的位置填写尚未明确的函数
变量了
## GOT全局偏移量表和PIC地址无关代码
前面说过，当可执行程序被加载到内存中后，会根据动态库加载到内存中的位置
完成对相应函数调用地址的填写，但是代码是存放在只读的代码段的啊？能修改吗?<br>
其实不能修改的。但是对于每个进程的地址空间来说，都有一张表，叫做全局偏移量表，
代码段的物理地址和动态库的物理地址都是确定的，所以它们之间的偏移量就是确定的，
因此完全可以通过cpu的相对寻址去找到动态库中每个函数的调用地址。在调用代码段的
代码时，先去查看GOT全局偏移量表，找到偏移量后，自然就能够调用到动态库
的函数了，这样，无论动态库被加载到物理内存的哪里，无论代码段被加载到物理内存
的哪里，所有进程都能够找到共享的一份动态库，这就叫做PIC地址无关代码

## 库间依赖和延迟绑定
有的动态库会会再次依赖其他动态库，这样依然可以通过PIC地址无关代码实现
，因为每个动态库都有自己的GOT表；同时，一个动态库中的函数是很多的，但是
并不是所有函数都会被调用，所以如果所有的函数都进行重定位的话，就会浪费很多时间
所以操作系统会采取类似懒启动的方式，当调用一个动态库中的函数时，再去更新GOT表中的
动态库中函数的物理地址

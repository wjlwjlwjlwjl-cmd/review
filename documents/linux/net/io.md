# 五种IO模型
## 1. 阻塞IO
<p>调用时如果没有就绪就一直等待，不再往下运行，知道IO时间就绪</p>

## 2. 非阻塞IO
<p>在程序使用系统调用后，如果IO事件不就绪，就返回继续往下执行代码，并采取轮询的方式反复询问操作系统内核IO事件</p>

## 3. 信号驱动IO
<p>当程序使用系统调用且IO事件不就续时，依然返回执行的等待，而现在不由程序自己去轮询是否IO就绪，而是由操作系统发送信号，告知IO事件就绪，由程序将内核中的IO数据拷贝到用户空间</p>

## 4. 多路转接
<p>当程序调用select时，阻塞在select函数，去等待多个套接字中的任意一个就位；就位后，阻塞解除，程序去内核中获取内容到用户空间处理</p>

## 5. 异步IO
<p>程序进行系统调用之后，如果没有数据报准备好，就返回继续执行程序；当数据准备好后，并且拷贝到用户空间之后，内核发送信号给程序，程序随后开始处理数据报</p>

## 多路转接之select
```cpp
int select(int nfds, fd_set *_Nullable restrict readfds,
                  fd_set *_Nullable restrict writefds,
                  fd_set *_Nullable restrict exceptfds,
                  struct timeval *_Nullable restrict timeout);
```
nfds是需要监听的文件描述符中最大的一个 + 1， readfds是一个位图，其中记录着需要监听读事件就绪的文件描述符，writefds记录着需要监听写事件就绪的文件描述符，而exceptfds记录着需要监听异常事件的文件描述符，操作位图的方法提供了这几种：FD_SET、FD_CLR、FD_ISSET、FD_ZERO，timeout表示等待时间，如果不设置相应内容就是一直等待
### select缺点
1. select每次使用时，需要自己设置文件描述符集合
2. 将fds从用户态拷贝到内核态和从内核态拷贝回用户态都是开销
3. 支持的文件描述符数量有上限
## 多路转接之poll
```cpp
int poll(struct pollfd *fds, nfds_t nfds, int timeout);
```
fds是记录需要监听的文件描述符相关信息的结构体数组，nfds是该数组的大小，timeout是等待时间<br>
pollfd中包含三个字段，一个是要监听的文件描述符，还有两个是要等待的事件和返回时查看的事件状态(events and revents)，时间包括POLLIN（对应的描述符可以读）POLLOUT(对应的描述符可以写) POLLERR（对应的描述符出错）
### poll的优点
1. 传参更加方便，直接传入结构体数组即可
2. 最大监听数目没有限制（虽然数量增加性能下降）
### poll的缺点
1. 将pollfd数组从用户态拷贝到内核态需要花销
2. 返回后，需要轮询pollfd数组来获取就绪状态
3. 也是因为轮询，如果监听大量文件描述符时，一般只有少量文件描述符就绪，所以轮询效率也会降低
## 多路转接之epoll
epoll可以认为是select和poll的改进版
### 调用接口
```cpp
int epoll_create(int size);
```
创建一个epoll结构体，其中存放有两个数据结构，一个是一棵红黑树，用来保存后续添加的所有监听事件的相关信息；另一个是一个双向链表，用来存放所有已经就绪的事件
```cpp
int epoll_ctl(int epfd, int op, int fd, struct epoll_event* event);
```
用来操作epoll事件，epfd是epoll_create返回的标识，op是操作的类型EPOLL_CTL_ADD、EPOLL_CTL_MOD、EPOLL_CTL_DEL，fd是要监听的描述符，event是事件类型，常用的包括EPOLLIN、EPOLLOUT、EPOLLERR，理解方式和POLL对应<br>
每使用epoll_create创建一个监听事件，都会在红黑树中添加一个节点，如果已经重复，就会以红黑树logN的时间复杂度完成遍历，确定是否添加；同时节点内部也和对应硬件的驱动程序存在绑定回调关系，如果时间就绪就会把节点自动添加到就绪的双向链表中
```cpp
int epoll_wait(int epfd, struct epoll_event* events, int maxevents, int timeout);
```
epfd是epoll的id，events用来带出已经就绪的事件；maxevents用来表示给出数组的大小；其大小不能超过epoll_create时指定的大小；timeout标识超时时间，0标识立即返回，-1标识永久阻塞；返回值如果是-1代表等待失败，如果是0代表超时
### epoll的优点
1. 操作简单
2. 不需要遍历的添加监听事件，而且可以在合适的时机添加进内核；同时从内核中返回时，不需要遍历返回的结果，直接查看epoll结构体中双向链表的大小即可，拷贝更加轻量而且数据利用效率更高
3. 可以同时监听的事件没有上限，而且不会像POLL监听事件增加而导致效率变低
### epoll的水平触发和边缘触发
#### 水平触发 lt
调用epoll_wait后，如果缓冲区中有需要读的数据，可以不全部读完，读完一部分后返回通知读事件就绪，后面继续读取，直到读完全部事件，只支持非阻塞读取
#### 边缘触发 et
调用epoll_wait后，只有一次读取机会，如果没读完，剩下的数据就不会再读取了，返回不会再告知事件就绪，下次再调用就直接阻塞等待；使用et模式的文件描述符必须是设置为非阻塞文件描述符，否则可能会陷入某种意义的死循环，比如以下场景：客户端向服务器写数据，服务器采取et模式没有全部读完，同时向客户端发送请求继续写入，但是客户端并没有接收到，导致服务端再次调用epoll_wait阻塞，因为客户端没有写数据（没读完的已经不算做就绪了），可客户端因为没有接收到服务端的应答也不会发送。所以将文件描述符设置为非阻塞，就可以采取非阻塞轮询的方式将缓冲区中的数据取完了
### 惊群效应
1. 在多线程多进程的环境中，可能会让多个线程、进程去等待epoll_wait，当事件就绪时，只有一个进程或者线程可以完成任务去处理事件，其他的线程或者进程等待失败，获得EAGAIN的errno，这会造成资源的浪费
2. 在多线程环境中，可以只让一个线程去等待该事件；在多进程环境中，可以分配一些负载较低的进程去完成这个过程，并且再加上锁，尽可能降低资源的浪费
